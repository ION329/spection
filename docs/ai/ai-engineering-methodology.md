# Spection — Spec-Driven Development Methodology

Version 2.1

Spection is a development methodology where the specification is the source of truth.
Code is a direct consequence of documentation, not the other way around.
AI acts as the documentation engine. The engineer acts as the system director.

---

## Core Principles

### Documentation is the system

All knowledge about the project lives in `docs/`.
This includes architecture, product definition, features, engineering specs, decisions, and workflows.
Nothing that matters lives only in someone's head or in a chat history.

### Code is a consequence

Implementation begins only after the specification is complete and validated.
A feature without a spec does not get coded.
A spec without engineering detail does not get tasked.
A task list without a clean context does not get executed.

### AI generates, engineers validate

AI is responsible for:
- generating documentation from prompts
- structuring specifications using templates
- expanding technical detail
- maintaining consistency across documents

The engineer is responsible for:
- directing the analysis
- validating every AI output before it advances to the next phase
- making architectural and product decisions
- approving the transition between phases

### Minimal start, organic growth

Repositories begin with only:
```
docs/
README.md
```

Code folders (`backend/`, `frontend/`, `services/`) appear only when implementation begins.
No folder is created speculatively.

### Strict context hygiene for implementation

Every implementation session must start with a clean context.
Load only the engineering spec and tasks file for the feature being coded.
Do not load product docs, architecture overviews, or unrelated specs into an implementation session.
Contaminated context degrades code quality and causes specification drift.

## Knowledge Structure

```
docs/
├── ai/       ← how AI collaborates in this project
├── spec/     ← full technical thinking of the system
├── public/   ← simplified documentation
└── lab/      ← experiments, research, and history
```

`docs/ai/` is the methodology engine. Its contents define how the AI behaves in this project.
`docs/spec/` contains all project-specific documentation generated by the AI and validated by the engineer.
`docs/spec/` is the only folder that changes as the project evolves.

---

## Phase Sequence

```
Discovery → Architecture → Feature Specs → Engineering Specs → Tasks → Implementation
```

Each phase produces a document. Each document must be validated by the engineer before the next phase begins.
No phase can be skipped. No phase can be reversed without revisiting all downstream documents.

---

## Roles

### Engineer (System Director)
- guides analysis and product thinking
- validates all AI outputs
- approves transitions between phases
- makes all architectural and product decisions
- is the only one who can mark a spec as ready for implementation

### AI (Documentation Engine)
- generates documentation from prompts
- asks clarifying questions before specifying
- follows templates exactly
- saves outputs to the specified paths
- never invents technologies, patterns, or requirements not explicitly approved
